Building a Production Multi-Agent System with MCP

Orchestrator pattern, FastMCP, LLM-driven tool selection, and the lessons learned keeping agents on the rails.

🕐 15 min read 📅 2025 🏷 MCP · Agents · OpenAI · Python

Artificial intelligence has moved well beyond chatbots that simply answer questions. The frontier today is agentic AI — systems that receive a goal, reason about what needs to be done, call external tools autonomously, and complete multi-step tasks without hand-holding.

I recently built a production-grade multi-agent system from scratch using Model Context Protocol (MCP), FastMCP, and OpenAI's o4-mini reasoning model. This article walks through the full journey: what MCP is and why it matters, how the system is architected, the beauty of LLM-driven tool selection, a real cross-agent workflow, and the critical lessons I learned about keeping agents on the rails.

01 What Is Model Context Protocol?

Before MCP, integrating an AI model with an external system required custom glue code for every combination of model and tool. Claude needed one connector for your database, another for Slack, another for your file system — and every other AI model needed the same connectors built separately. It was an N×M integration problem that did not scale.

Model Context Protocol is an open standard introduced by Anthropic in November 2024 and now adopted by OpenAI, Google DeepMind, and Microsoft. Think of it as USB-C for AI — a single universal connector between language models and the outside world.

The protocol defines a client-server architecture built on JSON-RPC 2.0. An MCP server exposes tools (functions the AI can call), resources (data it can read), and prompts (templates). An MCP client connects to one or more servers, discovers what is available, and routes the model's requests to the right place. Communication happens over stdio for local servers or Streamable HTTP for remote ones.

The elegant result: build one MCP server for your database, and every MCP-compatible AI client — Claude, your custom agent, Cursor, GitHub Copilot — can use it immediately.

02 The Orchestrator Pattern

For this project I chose the orchestrator pattern — the most scalable approach for multi-agent systems. Rather than building one large agent that knows everything, the system is divided into three distinct layers:

User types a task | v Orchestrator (OpenAI o4-mini) ├── decides which specialist(s) to delegate to ├── calls delegate_to_agent(agent_name, sub_task) | |---> LocalAgent ---------> server/server.py (FastMCP, stdio) | 9 tools: files, directories, SQLite database | └---> OpenMeteoAgent ---> npx open-meteo-mcp-server (stdio) 6 tools: forecast, archive, air quality, marine, elevation, geocoding

Receives the user goal. Reasons and delegates — never touches tools directly.

Layer 2a

Filesystem and SQLite expert. Connects to a local FastMCP server via stdio.

Layer 2b
OpenMeteoAgent
Weather and geocoding expert. Connects to an npm MCP server via npx.

The orchestrator has exactly one tool: delegate_to_agent(agent_name, sub_task). It never touches files, databases, or weather APIs directly. Its only job is to reason about the task, decide which specialist is responsible, and delegate with a clear sub-task string containing all the context the specialist needs.

03 Building the Local MCP Server with FastMCP

The local server is built using FastMCP — the high-level server framework that ships inside the official MCP Python SDK. Its key advantage is that it eliminates boilerplate entirely. Instead of manually writing JSON schemas, routing logic, and transport handling, you decorate Python functions:

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("local-fs-db-server")

@mcp.tool()
def db_write(key: str, value: str) -> str:
    """Insert or update a key-value note in the SQLite database.
    Uses UPSERT — if the key exists, its value is updated, not duplicated.

    Args:
        key:   Unique note key / label
        value: Note value / content
    """

    ...

FastMCP reads the function signature, generates the JSON schema from type hints, and uses the docstring as the tool description that the LLM reads when deciding what to call. The description is not just documentation — it is the primary signal the model uses to select the right tool.

Every one of the 9 tools is idempotent. db_write uses INSERT ... ON CONFLICT(key) DO UPDATE so running the agent repeatedly never creates duplicate rows. Directories use exist_ok=True. Files always overwrite. Agents will retry — the system must be resilient to that.

04 The OpenMeteo Agent: Zero Credentials

The OpenMeteoAgent connects to an external, community-built MCP server that requires no API key, no account, and no installation beyond Node.js. It runs entirely via:

StdioServerParameters(
    command="npx",
    args=["-y", "open-meteo-mcp-server"],
)

This is what the MCP ecosystem looks like in practice. A community developer packaged Open-Meteo's comprehensive weather API as an MCP server, published it to npm, and anyone can connect to it with three lines of configuration. The agent discovers the available tools at runtime by calling session.list_tools() — it does not need to know anything about the server implementation in advance.

05 LLM-Driven Natural Language Tool Selection

This is where the system becomes genuinely remarkable. No part of the code decides which tool to call. No if task contains "weather" then use openmeteo. The entire tool selection process is driven by the language model's natural language understanding.

Step 1
Task Arrives
User types a plain-English goal. Orchestrator receives it.
Step 2
o4-mini Reasons
Model reads task + agent descriptions, decides which specialist owns it.
Step 3
Delegate
Orchestrator calls delegate_to_agent() with a precise sub-task string.
Step 4
Tool Selection
Specialist LLM reads MCP tool list, picks tools, chains calls, returns result.

The LLM infers multi-step plans entirely from tool descriptions. Given the sub-task "get coordinates for Mumbai then fetch a 3-day weather forecast", the model recognises that geocoding must be called first to get latitude and longitude, then those coordinates must be passed to weather_forecast. Nobody wrote this logic. It emerges from the model's understanding of the tool descriptions and the natural dependencies between them.

06 A Real Cross-Agent Workflow

Let me walk through a concrete end-to-end example that shows both agents working together.

User input: "Get the 3-day weather forecast for Mumbai and save each day to the local database"

Orchestrator step 1 — Recognises this spans two domains: weather retrieval and local storage. Delegates to OpenMeteoAgent first with a precise sub-task. The agent calls geocoding(name="Mumbai") to get coordinates, then weather_forecast with those coordinates. It returns three days of structured weather data.

Orchestrator step 2 — Receives the weather data and delegates to LocalAgent, passing the actual data inline in the sub-task string. The LocalAgent calls db_write three times with unique date-based keys — each using UPSERT, so re-running tomorrow updates rather than duplicates.

Result: Three rows live in a local SQLite database. No Python scripts. No SQL written by hand. No shell commands. The LLM understood the intent, selected the right tools in the right sequence, and coordinated data transfer between two completely separate agents — all from a single sentence of plain English.

07 The Critical Lesson: System Prompts as Control

During development I encountered a fundamental failure mode. When the LocalAgent was asked to "save weather data to the database", it occasionally responded by generating a Python script using sqlite3 directly, then suggesting the user run it manually. The LLM defaulted to what it knows best — writing code — rather than using the MCP tools it had been given.

The fix was not in the code. It was in the system prompts.

⚑ Key architectural lesson

Positive guidance alone is not enough. You need explicit prohibitions against the failure modes you have observed. Vague prompts produce creative but wrong behaviour. Precise, specific, prohibitive prompts produce reliable, predictable tool use. The system prompt is your primary control mechanism.

I added explicit prohibitive rules to both the agent and orchestrator prompts — for example, the LocalAgent prompt states: "You ONLY use the tools listed above. You never write Python scripts, shell commands, SQL statements, or any code as your response."

The same principle applies to tool descriptions on the server side. A vague description like "saves data" will cause the model to misuse the tool or skip it. A specific description like "Insert or update a key-value note using UPSERT — safe to call multiple times" gives the model exactly what it needs to make the right decision.

09 Extensibility: Adding a New Agent in Two Steps

The architecture is designed to grow. Adding a third specialist — a GitHub agent, a Slack agent, a PostgreSQL agent — requires exactly two changes:

Step 1: Create agents/github_agent.py extending BaseAgent, set server_params to point at the MCP server package.

Step 2: Register it in orchestrator.py with a one-line description.

The orchestrator automatically includes the new agent in the LLM's decision context. No changes to existing agents. No changes to BaseAgent. No changes to session management. The system self-extends.

The growing ecosystem of community-built MCP servers — covering GitHub, Slack, Notion, Jira, PostgreSQL, Stripe, and hundreds more — becomes a library of plug-and-play capabilities any agent can connect to with a few lines of configuration.

10 Conclusion

Building this system reinforced something I believe deeply: the hard part of multi-agent AI is not the code. The hard part is the design of responsibilities, the clarity of tool descriptions, and the precision of system prompts.

Get those right and the LLM does exactly what you intend — selecting the right tools in the right sequence, coordinating across agents, and handling complex multi-step tasks from plain English with no hardcoded logic anywhere in the pipeline. Get them wrong and you will spend your time chasing hallucinated scripts and misrouted delegations.
