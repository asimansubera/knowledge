Indexing in the Age of Vectors

From B-tree execution plans to HNSW graph traversal — how vector databases make similarity search fast at scale.

12 min read 2025 HNSW · RAG · Vector DB

Having spent years working with database technologies, I've seen performance optimization evolve dramatically — from tuning Oracle execution plans to architecting modern data platforms. One challenge that consistently fascinates me is indexing: how do you make a query engine return results fast without reading everything?

In traditional relational databases like Oracle, PostgreSQL, or SQL Server, B-tree and bitmap indexes are the workhorses. They work beautifully for structured, low-dimensional data — filtering rows by customer ID, date ranges, or categorical values. The query engine uses these indexes to touch the least amount of data possible, keeping latency low even at scale.

But vector databases break all of those assumptions.

01 The Core Problem with Vector Search

Vector databases store high-dimensional numerical representations of data — embeddings from LLMs, image encodings, audio fingerprints. A single vector might have 768, 1536, or even 3072 dimensions. Similarity search in this space means finding vectors that are "close" to a query vector using distance metrics like cosine similarity or Euclidean distance.

B-tree and bitmap indexes are fundamentally incompatible with this problem. They are designed for exact lookups and range scans in low-dimensional space — not for measuring geometric proximity across thousands of dimensions. Without a purpose-built index, a vector database has no choice but to perform a full scan of every vector in the collection, comparing each one against the query. At millions of vectors and high QPS, that latency becomes completely unacceptable.

This problem is especially acute in Retrieval-Augmented Generation (RAG) pipelines, where an LLM retrieves relevant context from a vector store at query time. A slow or inaccurate retrieval step directly degrades the quality and responsiveness of the AI response — making efficient vector indexing not just a performance concern, but a product requirement.

02 Enter HNSW — Hierarchical Navigable Small World

HNSW is a graph-based indexing algorithm specifically designed for approximate nearest neighbor (ANN) search in high-dimensional spaces. I recently explored Qdrant, a vector database built with HNSW at its core, and the design choices are worth understanding in depth.

When a vector is ingested, Qdrant builds an HNSW graph in memory organized in multiple layers:

Layer N (top)
Sparse Graph
Fast long-range traversal — quickly narrows the search space
Layer 1–N−1
Intermediate
Progressive refinement — descends toward the target neighborhood
Layer 0 (bottom)
Dense Graph
Precise local search — identifies the true nearest neighbors

This hierarchical structure means the engine never scans the entire dataset — it navigates intelligently, reducing search time to the millisecond range.

03 Quantization: Keeping More Data in RAM

One of Qdrant's most effective performance techniques is quantization. Raw vector data is typically stored as FLOAT32 on disk. Qdrant can compress these to INT8 through scalar quantization, reducing the memory footprint by approximately 4×. This matters enormously because the HNSW graph and the vectors it references need to live in RAM for fast access — avoiding page faults during search. For raw vector data, Qdrant uses memory-mapped files (mmap), leveraging Linux's page cache to manage disk reads efficiently.

The search workflow is a two-phase approach:

Phase 1
ANN via INT8
HNSW finds approximate candidates using the quantized index in RAM
Phase 2
Re-score via FLOAT32
Candidates re-ranked against full-precision vectors for accuracy
Output
Final Results
~95% recall at <50ms — even at 1,000+ QPS with concurrent ingestion

04 Distributed Architecture: Shards & Segments

A single machine cannot hold an entire large vector collection. Qdrant addresses this through distributed deployment using shards and segments.

A shard is the fundamental unit of distribution — both the vector data and its associated HNSW graph live together within a single shard. This co-location is intentional: it avoids the overhead of cross-node graph traversal during search. Segments within a shard allow incremental indexing, so freshly ingested vectors can be searched immediately while background indexing builds their HNSW structure.

Payload Indexes: Beyond HNSW, Qdrant supports payload indexes on metadata fields (user ID, category, timestamp, region). These allow the query engine to route traffic to specific shards rather than broadcasting to every node — critical at scale. Without them, every search becomes a fan-out across the entire cluster, multiplying latency and infrastructure cost.

05 HNSW Tuning: Recall vs. Latency

HNSW exposes configuration parameters that let you tune the graph structure for your specific workload:

Parameter What it controls Trade-off
m Bi-directional links per node Higher → better recall, more memory & build time
ef_construct Candidate list size during index build Larger → more accurate graph, longer build time
ef Candidate list size during search Higher → better recall, higher query latency

These parameters give engineers direct control over the recall-latency trade-off, which is essential when designing for different SLAs — a real-time recommendation system has very different requirements than an overnight batch similarity job.

This is particularly relevant today as RAG has become the dominant pattern for grounding LLMs in private or real-time data. In a RAG system, every user query triggers a vector similarity search to fetch the most relevant document chunks before the LLM generates a response. The entire experience — accuracy, latency, cost — hinges on how well that retrieval step performs. HNSW-backed vector databases like Qdrant are what make production RAG systems viable at scale.

06 The Bigger Picture

What makes HNSW compelling is not just its speed — it's that it brings the same kind of index-aware query optimization we've relied on in relational databases for decades, now applied to the geometry of high-dimensional space. The principles are familiar: avoid full scans, organize data structures to narrow the search space early, and keep hot data close to the compute.

If you're building AI-powered applications, RAG pipelines, or semantic search systems, understanding how your vector database indexes and retrieves data is as important as it ever was in the RDBMS world. The tools have changed — the discipline hasn't.
